# 示例 Product Spec：Job 状态语义对齐

- 需求一句话：
  - 统一 `Job.Create -> syncArgo -> 外部事件回写` 之间的状态语义，避免把“已发起同步”误判为“发布成功”。
- 目标：
  - 明确 `POST /api/v1/jobs` 的成功语义。
  - 明确 `Pending`、`Syncing`、`SyncFailed` 与运行终态的边界。
  - 让 `Job.status`、日志、Argo 侧标签在语义上可对齐、可验收。
  - 为后续代码改造提供单一 spec，避免在 `pkg/api`、`pkg/service`、文档之间各说各话。
- 非目标：
  - 不重做 Argo/Tekton 客户端。
  - 不新增 Job 类型。
  - 不讨论前端展示。
  - 不直接改动线上集群或外部系统。
- 影响资源：
  - `Job`
  - `Application`
  - `Manifest`
- 影响模块：
  - `pkg/api/job.go`
  - `pkg/service/job.go`
  - `agents/reference/job.md`
  - `agents/patterns/status-flow.md`
- 关键行为：
  - `POST /api/v1/jobs` 返回 200 + `id` 仅表示 Job 已落库，且同步请求流程已被触发。
  - `Pending` 表示记录刚建立。
  - `Syncing` 表示本地已进入向 Argo 发起同步的流程，不能等价为发布成功。
  - `SyncFailed` 仅表示本地发起同步失败。
  - `Running`、`Succeeded`、`Failed`、`RollingBack`、`RolledBack` 应优先由外部事件驱动，与 ADR 001 保持一致。
- 状态流 / 数据流：
  - `POST /api/v1/jobs`
  - `JobService.Create`
  - Mongo 创建 Job 记录并初始化 `Pending`
  - 本地状态推进到 `Syncing`
  - `syncArgo`
  - 若本地调用失败，回写 `SyncFailed`
  - 若本地调用成功，后续运行态与终态由外部事件回写
- 外部依赖与限制：
  - Argo CD：由 `pkg/service/job.go` 调用。
  - Mongo：`Job.status` 的持久化载体。
  - `devflow-common`：模型与外部客户端定义不在当前仓库内。
  - 当前仓库中未直接展示完整的外部事件消费链路，因此终态回写只能先作为约束写入 spec。
- 验收基线：
  - 文档层面对每个状态的来源与含义有单一表述。
  - `Job.Create` 的成功语义不再与“发布完成”混淆。
  - Evaluator 能据此检查 handler、service、日志、状态字段是否一致。
  - 若后续改代码，测试至少覆盖创建成功、同步失败、终态依赖外部事件这三类场景。
- 风险 / 未知项：
  - 当前 `pkg/service/job.go` 在调用 `updateStatus(..., Syncing)` 后未同步更新内存中的 `job.Status`，日志语义可能与持久化状态不一致。
  - 当前 Argo 侧标签写入 `status=Running`，与 Mongo 中 `Syncing` 语义存在潜在偏差。
  - `go test ./...` 当前基线要求先执行 `go mod tidy`。
