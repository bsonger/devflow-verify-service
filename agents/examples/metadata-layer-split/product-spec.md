# 示例 Product Spec：Devflow 元数据层拆分

- 需求一句话：
  - 把 `devflow` 收敛成元数据层，只保留资源模型、状态与编排意图；Tekton 和 Argo 的实际执行能力拆成独立服务。
- 目标：
  - 明确哪些职责保留在 `devflow`。
  - 明确哪些职责搬出到 Build Executor / Deploy Executor。
  - 给后续代码拆分设定最小迁移顺序，避免一次性重构过大。
  - 保持现有 `Application`、`Manifest`、`Job` 作为统一元数据模型。
- 非目标：
  - 不在本轮决定消息中间件选型。
  - 不重构前端或客户端协议。
  - 不立即删除全部 Tekton / Argo 代码。
  - 不直接改线上部署方式。
- 影响资源：
  - `Application`
  - `Manifest`
  - `Job`
  - `Configuration`
- 影响模块：
  - `pkg/config`
  - `pkg/service/manifest.go`
  - `pkg/service/job.go`
  - `pkg/service/argo.go`
  - `agents/`
- 保留在元数据层的职责：
  - `Application` / `Manifest` / `Job` / `Configuration` 的 CRUD 与查询
  - active manifest 绑定
  - Job / Manifest 状态读模型
  - 接收外部事件并写回 Mongo
  - 发起 build / deploy intent
- 拆出的职责：
  - Build Executor：PVC、PipelineRun、TaskRun、Tekton 状态采集
  - Deploy Executor：Application create/update、sync、Argo 状态采集
  - 如有需要，独立 Event Ingestor 负责把外部事件标准化后写回元数据层
- 状态流 / 数据流：
  - 客户端请求 `devflow`
  - `devflow` 写入 `Application` / `Manifest` / `Job`
  - `devflow` 产生 build 或 deploy intent
  - Executor 消费 intent 并调用 Tekton / Argo
  - Executor 或事件采集器把外部状态回写到 `devflow`
  - 客户端统一从 `devflow` 查询状态
- 外部依赖与限制：
  - 当前代码启动时强依赖 kubeconfig、Tekton、Argo 客户端初始化。
  - `devflow-common` 中的 client/model 定义不在本仓库内，接口抽象需要留意兼容性。
  - 当前仓库还没有显式的 intent queue / event endpoint 定义。
- 验收基线：
  - 有一份清晰的边界清单，说明每个模块保留或迁移的理由。
  - 有一份最小迁移顺序，不要求一次改完。
  - Evaluator 能判断某个改动是在“收敛为元数据层”还是又把执行逻辑拉回来了。
- 风险 / 未知项：
  - 若先拆执行逻辑、后补事件回写，会导致状态断层。
  - 若消息协议先行但模型边界不清，会把耦合从代码移动到消息体。
  - 当前 `pkg/service/argo.go` 的事件处理仍很薄，状态回写路径尚未闭环。
